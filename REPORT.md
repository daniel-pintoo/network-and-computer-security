# T50 ChainOfProduct

## 1. Introduction

This project is developed in the context of the **Chain of Product (CoP)** scenario, where multiple companies collaborate to produce and deliver a product, from raw materials to the final customer. Because these companies often do not fully trust each other, and trade secrets must be protected, the smartphone company **Samchunga** decided to rely on a public third-party service to store, validate, and support the controlled sharing of Delivery-vs-Payment (**DvP**) transaction documents.

The original DvP transaction is represented as a JSON object containing the fields: `id`, `timestamp`, `seller`, `buyer`, `product`, `units`, and `amount`. Since this document contains sensitive business information, Samchunga requires protection mechanisms that guarantee confidentiality, integrity, authentication, and transparent sharing verification, in alignment with scenario requirements **SR1–SR4**.

The project is divided into three components:

- **Secure Documents** — design a library to protect the original JSON  
- **Infrastructure** — servers and network environment  
- **Security Challenge** — additional requirements that extend the system

## 2. Project Development

### 2.1. Secure Document Format

#### 2.1.1. Design

| Requirement                     | How it's satisfied                                                                |
|---------------------------------|-----------------------------------------------------------------------------------|
| **SR1: Confidentiality**        | AES encryption + wrapped keys (only recipients with private keys can decrypt)     |
| **SR2: Authentication**         | A sharer must already hold access and create a signed access entry to disclose to another recipient |
| **SR3: Integrity**              | The seller signs the protected document, and `check()` verifies the seller signature |
| **SR4: Integrity of Sharing History** | Sharing records are signed and stored in the access list; the current verification path focuses on seller-signed entries |

### Protected Document Structure

The third party stores a single JSON structure like this:

```json
{
  "ciphertext": "...",        
  "iv": "...",

  "wrapped_keys": [
    {
      "recipient": "Seller",
      "wrapped_key": "..."
    },
    {
      "recipient": "Buyer",
      "wrapped_key": "..."
    }
  ],

  "signatures": {
    "seller_signature": "..."
  },

  "access_list": [
    {
      "recipient": "CompanyA",
      "wrapped_key": "...",
      "signed_by": "Seller",
      "timestamp": 17663363700,
      "signature": "..."      
    }
  ],

  "metadata": {
    "transaction_id": 123,
    "version": 1
  }
}
```


#### 2.1.2. Implementation

The secure document functionality was implemented as a custom Java library, designed to protect transaction documents in the Chain of Product scenario by ensuring confidentiality, integrity, authenticity, freshness, and auditability.

The implementation follows a hybrid cryptographic approach and relies exclusively on standard, well-established cryptographic primitives provided by the Java Cryptography Architecture (JCA).

#### Programming Language and Cryptographic Libraries

The solution was implemented in Java, using the Java Cryptography Architecture (JCA) and its standard providers. No custom cryptographic algorithms were implemented.

The following algorithms were used:

- AES-256-GCM for symmetric encryption of document contents;

- RSA-2048 with OAEP padding for key wrapping;

- RSASSA-PSS for digital signatures;

- SecureRandom for cryptographically secure random number generation.

#### Key Material and Distribution

The secure document system relies on the following cryptographic keys, generated and distributed as described below:

Seller Key Pair (RSA-2048):
Generated locally by the seller.
The private key is used to digitally sign protected documents.
The public key is distributed to buyers and other authorized recipients to allow signature verification.

Buyer Key Pair (RSA-2048):
Generated locally by the buyer.
The private key is used to unwrap the document’s symmetric key.
The public key is distributed to the seller to allow key wrapping.

Recipient Key Pairs (RSA-2048):
Each additional recipient generates their own key pair locally.
Public keys are distributed to the seller (or obtained through the application infrastructure) to enable controlled document sharing.

Document Session Key (AES-256):
Generated by the seller for each document.
Used to encrypt the document payload.
Individually wrapped using the public key of each authorized recipient.

In the current prototype, private keys are generated locally by each participant but are uploaded to the API over HTTPS when protect, share, or unprotect operations are requested. They are not stored inside the protected document format itself.

#### Choice of Cryptographic Algorithms

The cryptographic algorithms were selected based on current best practices and standard recommendations.

- AES-256-GCM was chosen because it is a widely recommended authenticated encryption scheme, providing confidentiality and integrity in a single operation and being commonly used in secure systems.

- RSA-OAEP was used for key wrapping as it is the recommended padding scheme for RSA encryption, providing strong security guarantees and being suitable for encrypting symmetric keys. The implementation prefers SHA-256-based OAEP and falls back to SHA-1-based OAEP if provider support is unavailable.

- RSASSA-PSS was selected for digital signatures because it is the recommended RSA-based signature scheme in modern cryptographic standards, offering improved security compared to older padding methods. The implementation prefers RSASSA-PSS and falls back to `SHA256withRSA` when necessary for compatibility.

All selected algorithms are supported natively by the Java Cryptography Architecture.

#### Secure Document Format

Each protected document is composed of:

- An encrypted payload containing the original JSON document (AES-256-GCM);

- Metadata including timestamps and nonces to ensure freshness;

- A digital signature generated by the seller (RSASSA-PSS, with compatibility fallback in the current implementation);

- A list of access entries, each containing a wrapped symmetric key for an authorized recipient (RSA-OAEP).

This structure allows the document to be encrypted only once while supporting secure and auditable sharing with multiple recipients.

#### Example of Document Protection

Plaintext Document

```json
{
  "id": 123,
  "timestamp": 17663363400,
  "seller": "Ching Chong Extractions",
  "buyer": "Lays Chips",
  "product": "Indium",
  "units": 40000,
  "amount": 90000000
}
```
Protected Document

```json
{
  "metadata": {
    "transaction_id": 123,
    "version": 1
  },
  "ciphertext": "Base64(AES-GCM encrypted payload)",
  "iv": "Base64(12-byte IV)",
  "timestamp": 1736951530000,
  "nonce": "Base64(random nonce)",
  "signatures": {
    "seller_signature": "Base64(signature)"
  },
  "wrapped_keys": [
    {
      "recipient": "Seller",
      "wrapped_key": "Base64(RSA-OAEP wrapped AES key)"
    },
    {
      "recipient": "Buyer",
      "wrapped_key": "Base64(RSA-OAEP wrapped AES key)"
    }
  ],
  "access_list": []
}
```

The document payload is encrypted once using AES-GCM, while access control is enforced by distributing individually wrapped session keys to authorized recipients.

#### Implementation Challenges and Solutions

- Key Distribution and Access Control:
A major challenge was enabling secure sharing with multiple recipients without re-encrypting the entire document. This was addressed by encrypting the document once with a symmetric key and wrapping that key individually for each authorized recipient.

- Integrity and Authenticity:
To ensure recipients could verify the origin and integrity of documents, the seller digitally signs the protected document with their private key, and verification is performed using the seller’s public key extracted from a CA-signed certificate.

- Freshness and Replay Protection:
Timestamps and nonces were included in the protected document to provide freshness metadata and to bind these fields to the seller’s signature. The current verification flow checks their presence and signed integrity; stronger replay protection would require additional stateful validation against previously seen documents.

- Auditability:
Access entries explicitly record which entities received access to a document and who gave them the access, providing non-repudiation (by signing the access entry with their private key) and allowing the seller and buyer to verify disclosure decisions.

#### 2.2.1. Network and Machine Setup

We built a small isolated infrastructure with 5 Kali Linux VMs on VirtualBox:

<cop-client> (10.10.10.100/24, front-net): used as the user client. It only needs a browser and curl; no services are exposed.

<cop-app> (10.10.10.10/24 front-net, 10.10.20.10/24 back-net): application server running the FastAPI API in Python, the Java secure-document library (via CLI), and nginx for HTTPS termination. We chose FastAPI/Python because we were more familiar with it and it integrates easily with the existing Java library. Java is used only for cryptographic operations, reusing the library from the course labs.

<cop-db> (10.10.20.20/24, back-net): PostgreSQL 17 database server storing protected documents. We selected PostgreSQL because it has good TLS support, native JSON types, and simple configuration for host-based access control.

<cop-group> (10.10.10.20/24, front-net): group-server for dynamic group management (members and groups, with certificates), implemented as a separate FastAPI microservice to keep group logic independent from the main API.

<cop-ca> (10.10.10.30/24, front-net): internal Certificate Authority server that issues X.509 certificates for all partners and exposes `/root-ca` and `/sign` endpoints used by clients and servers.

##### Network layout:

<front-net> (10.10.10.0/24): connects cop-client, cop-app, cop-group, and cop-ca. Only API, group, and CA endpoints are reachable here.

<back-net> (10.10.20.0/24): connects cop-app and cop-db. Only database traffic flows here.

Each VM also has a NAT adapter for apt/updates; default routes are via NAT.

##### Firewalls:

UFW is used on all VMs for simple and explicit rules (documented in FIREWALL_TLS_SETUP.md).

<cop-client>: default deny incoming, allow outgoing; no open ports.

<cop-app>: deny incoming except API port (443) from 10.10.10.0/24; outgoing allowed (including DB connections on back-net).

<cop-db>: deny incoming except 5432/tcp from 10.10.20.10; outgoing allowed.

<cop-group>: deny incoming except 8001/tcp from front-net; outgoing allowed.

<cop-ca>: deny incoming except 8002/tcp from front-net; outgoing allowed.

#### 2.2.2. Server Communication Security

We secured server communications at two layers: network isolation (front-net/back-net + UFW) and TLS on the relevant channels.

##### <Front-net> (client ↔ app, client ↔ group-server, app ↔ group-server, client/app ↔ CA):

HTTPS is provided by nginx on cop-app listening on 10.10.10.10:443.  
Uvicorn (FastAPI) runs on 127.0.0.1:8000, and nginx proxies 443 → 127.0.0.1:8000.

We created an internal CA on cop-app (ca.key/ca.crt), issued an app.crt/app.key for 10.10.10.10, and configured nginx to use them.

ca.crt is installed on cop-client so curl and the browser can validate https://10.10.10.10 without warnings.

tcpdump on cop-app’s eth0 at port 443 shows only TLS-encrypted traffic; HTTP payloads and keys are not visible on the wire.

Group-server runs HTTP on 10.10.10.20:8001 on front-net. We don't think HTTPS is a better option because there is no need to encrypt the data this microservice transmits. The objective of this microservice is to be a centralized repository of user information where other users can grab the public keys of their business partners, so the attacker isn't able to gain more information than a mere normal user.

cop-app uses GROUP_SERVER_URL to call POST /api/groups/{group_name}/resolve and obtain current group members and their registered certificates (certificate_pem). For each member, the application validates the certificate against the Root CA, extracts the public key, and uses it to wrap the AES session key.

cop-client (and other client VMs) can also access the group-server API (e.g. via http://10.10.10.20:8001/docs) to register members and groups. UFW on cop-group restricts access to the front-net, where all client VMs live.

##### <Back-net> (app ↔ db):

PostgreSQL on cop-db is bound only to 10.10.20.20 and localhost.
TLS is enabled in postgresql.conf using db.crt/db.key signed by the same internal CA, and ssl_ca_file points to /etc/postgresql/17/main/tls/ca.crt.

pg_hba.conf requires hostssl appdb appuser 10.10.20.10/32 md5.

On cop-app, the API connects with sslmode=require (and can use sslrootcert=ca.crt to verify the server). \conninfo confirms SSL Connection: true (TLSv1.3).

UFW on cop-db only allows 5432 from 10.10.20.10, so only the application server can open DB connections.

##### Keys at the start and distribution:

Document cryptography keys:  
We generate RSA keypairs for seller, buyer, and third parties (e.g. CompanyA, CompanyB) using generate-client-keys.sh.

Private keys remain on the corresponding clients (or test VMs) and are uploaded via HTTPS when needed (protect, unprotect, share).

For each partner, a CSR is generated and submitted to the CA service on cop-ca. The resulting CA-signed X.509 certificate is then:

- registered with the group-server via POST /api/members/upload (stored as `certificate_pem` in members.json);
- provided to the API (as seller/buyer/new recipient certificates) when protecting or sharing documents.

At the server side, the application validates these certificates against the Root CA, extracts the public keys (SubjectPublicKeyInfo bytes), and passes them into the Java library for AES key wrapping and signature verification.

TLS CA keys:  
The internal CA keypair is generated and stored on cop-ca. Its private key never leaves that VM.

app.crt/app.key and db.crt/db.key are created and deployed to nginx on cop-app and PostgreSQL on cop-db, respectively.

ca.crt is copied to cop-client (system trust store + browser) and to cop-db (for DB TLS), and can also be used by the API as sslrootcert.

Group certificates and public keys:  
Group members’ certificates are registered centrally in the group-server (certificate_pem). When a group is resolved, cop-app fetches the member list, validates each certificate against the Root CA, extracts the public keys, and uses them to wrap the document’s AES key for each member.

### 2.3. Security Challenge

#### 2.3.1. Challenge Overview
The Security Challenge A introduces the need to simplify transaction disclosure by allowing documents to be shared with groups of partners, instead of requiring the seller to manually disclose to each individual recipient.

In the original design, transaction disclosure was performed only on a per-partner basis through the share() operation, which becomes laborious and error-prone as the number of partners grows. The new requirement allows a transaction to be disclosed to a group in a single operation, ensuring that all partners belonging to that group at the moment of disclosure can read and validate the transaction information.

Additionally, the challenge requires the existence of a separate server that dynamically tracks groups and their members, enforcing the disclosure rules valid at the moment of sharing. This requirement impacted the original design by introducing:
  1. A new dedicated server (Group Server) responsible for group and membership management
  2. A snapshot mechanism to capture group membership at the exact moment of disclosure
  3. A new API endpoint (share-to-group) that integrates with the Group Server
  4. Temporal access control ensuring that members who join a group after disclosure cannot access previously shared documents
     
The cryptographic mechanisms for document protection (AES-256-GCM encryption, RSA signatures, and freshness protection) remain unchanged. The extension operates exclusively at the access control layer, by dynamically resolving group membership at disclosure time and creating individual wrapped keys for each partner that belongs to the group at that moment.

#### 2.3.2. Attacker Model

#### Trust Assumptions 

Fully Trusted Entities

- Seller and Buyer: Trusted to correctly generate, protect, and sign transaction documents using their private keys. They are authorized to share documents with other partners and must not leak their private keys. 
- Secure Document Library: Trusted to correctly implement cryptographic operations (encryption, signing, key wrapping) according to specifications. 

Partially Trusted Entities 

- Group Management Server: Trusted to correctly manage groups and members and maintain membership records, but not trusted with access to document contents or private keys. Could potentially provide false membership information, but this is mitigated by cryptographic verification (members still need valid private keys to decrypt). 
- Application Server (Main API): Trusted to correctly enforce protocol logic and invoke the secure document library. In the current prototype it temporarily handles plaintext documents and uploaded private keys over HTTPS during protect, share, and unprotect operations. 
- Database Server: Trusted for availability and storage integrity, but not trusted for confidentiality, since stored documents are encrypted. The server can read encrypted data but cannot decrypt it. 

Untrusted Entities

- External Attackers: Network adversaries with capabilities to eavesdrop, intercept, or modify traffic. 
- Unauthorized Partners: Any entity not explicitly included in a transaction's disclosure list or group membership at the moment of sharing. 
- Compromised or Malicious Partners: Partners who may attempt to forge shares, replay old transactions, or gain unauthorized access. 

#### Attacker Capabilities and Limitations

What the Attacker can Do:

- Network Attacks: Intercept, replay, or modify network traffic between clients, servers, and the Group Server. Perform man-in-the-middle attacks on unprotected communication channels. 
- Data Access: Query the Group Server for group membership information. 
- Impersonation Attempts: Attempt to impersonate partners without possession of valid cryptographic keys. Submit malicious requests to the Group Server (e.g., unauthorized group modifications). 
- Social Engineering: Convince legitimate partners to share documents inappropriately. 

What the Attacker can't Do: 

- Cryptographic Attacks: Cannot decrypt protected documents without possessing the corresponding valid private key. Cannot forge valid digital signatures without access to private signing keys. Cannot break AES-256-GCM encryption or RSA-2048 cryptographic protections (assuming standard cryptographic assumptions). 
- Temporal Access Control Violations: Cannot retroactively gain access to documents by joining a group after the moment of disclosure. Cannot alter or forge group membership snapshots that were captured at the time of disclosure. 
- Replay and Tampering of Freshness Fields: An attacker cannot alter signed timestamp/nonce fields without invalidating the seller signature. Preventing re-submission of an unchanged older document would require additional replay-tracking state beyond the current prototype. 
- Key Recovery: Cannot extract private keys from encrypted documents or signatures. Cannot derive AES session keys from wrapped keys without the corresponding RSA private key. 

#### Security Assumptions

This attacker model relies on the following standard cryptographic security assumptions:

- Key Secrecy: Private keys are generated client-side and are assumed not to be leaked or compromised. In the current prototype they are transmitted to the API over HTTPS for some operations, so transport security and server trust are part of the security assumptions. 
- Cryptographic Strength: AES-256-GCM, RSA-2048, and RSASSA-PSS provide adequate security against cryptanalytic attacks. 
- Random Number Generation: Nonces, IVs, and session keys are generated using cryptographically secure random number generators. 
- Implementation Correctness: Cryptographic libraries (Java Cryptography Architecture) are correctly implemented and do not contain exploitable vulnerabilities.
- Timestamp Synchronization: Reasonable time synchronization exists between parties for freshness validation.

#### 2.3.3. Solution Design and Implementation

To address Security Challenge A, the original system was extended, preserving the existing secure document protection mechanisms while introducing group-based disclosure through a dedicated Group Management Server.

The solution introduces a clear separation of responsibilities between document protection, access control, and group management, ensuring that the cryptographic core of the system remains unchanged.

#### Group Management Server

A separate Group Management Server was introduced to manage groups of partners and their memberships. This server is responsible for:

Creating and managing groups;

Registering partners and associating them with their CA-signed certificates (certificate_pem);

Validating certificates against the Root CA before storage;

Dynamically resolving group membership at the moment of disclosure;


The Group Server does not perform any cryptographic operations and never accesses document contents or private keys. Its role is strictly limited to managing group metadata.
To improve availability while keeping the implementation lightweight, the Group Server uses persistent JSON-based storage. Group and member information is stored on disk and loaded into memory at startup, ensuring that group definitions survive server restarts without requiring a full database system. This design choice simplifies deployment while remaining fully compatible with the security requirements of the challenge.

#### Key Distribution

The introduction of group-based disclosure does not modify the original key distribution model. No cryptographic keys are associated with groups themselves. Instead, each partner maintains an individual RSA-2048 public/private key pair generated locally.

Each partner obtains a CA-signed X.509 certificate from the CA service (cop-ca) and registers this certificate with the Group Management Server. The Group Server never stores or processes private keys and does not perform any cryptographic operations beyond certificate validation.

When a document is shared with a group, the group is dynamically resolved to its current list of members. For each resolved member, the Application Server:

- validates the stored certificate against the Root CA;
- extracts the public key from the certificate;
- individually wraps the document’s symmetric AES session key using that RSA public key (RSA-OAEP).

Only partners possessing the corresponding private keys are able to unwrap the session key and decrypt the document content.

#### Security properties:

  - Private keys are generated by each partner and are not embedded in the protected document format. In the current prototype, they are transmitted to the API over HTTPS when required for cryptographic operations.

  - Public keys are distributed in encoded form, avoiding dependencies on filesystem-based key references.

  - Each partner’s access is cryptographically independent; no shared group-level key exists.

#### Group-Based Disclosure Workflow and Message Exchanged

When a protected document is shared with a group, the following sequence describes both the group-based disclosure workflow and the message exchange between system entities:

    1. Disclosure Request:
    An authorized partner (e.g., the seller) requests disclosure of a protected document to a specific group using the POST /api/documents/{id}/share-to-group endpoint.

    2. Group Resolution:
    The Application Server contacts the Group Management Server through the /api/groups/{group_name}/resolve endpoint to obtain the current group membership.

    3. Snapshot Creation:
    The Group Management Server returns a snapshot containing:
      - A unique snapshot identifier,
      - The group name and resolution timestamp,
      - The list of members belonging to the group at that moment, including their registered certificates (certificate_pem).
      
    4. Certificate Validation and Individual Key Wrapping:
    The Application Server treats each member in the snapshot as an individual recipient. For each one, it validates the member’s certificate against the Root CA, extracts the RSA public key from the certificate, and invokes the existing document sharing mechanism. The document’s AES session key is wrapped separately for each member using their respective RSA public key.

    5. Document Update:
    The updated protected document is stored in the database with new access entries and wrapped keys for the resolved members. The API response includes the snapshot identifier and resolution timestamp returned by the Group Server.

    6. Temporal Enforcement:
    Only partners present in the snapshot at the moment of disclosure receive wrapped keys. Partners who join the group after this moment cannot access previously shared documents, as they were not included in the snapshot and therefore lack the required wrapped session key.

#### 2.3.4. Mapping to Security Requirements (SR1–SR4)

- **SR1 – Confidentiality**:  
  Confidentiality is ensured by encrypting every transaction document with AES‑256‑GCM and by wrapping the symmetric AES key individually for each authorized partner using RSA‑OAEP. The `unprotect()` operation will fail (no AES key can be unwrapped) if an attacker or unauthorized partner does not own a corresponding private key, which we demonstrate in the usage tests.

- **SR2 – Authentication of Sharing Operations**:  
  Only principals that already have access to a document can invoke `share()`: the Java library first proves that the sharer can unwrap at least one existing wrapped key, and only then it allows the creation of a new access entry. This means the current implementation authorizes any existing holder, not only the original seller or buyer. Each access entry is signed with the sharer’s private key and records who shared, with whom, and when.

- **SR3 – Integrity of the Protected Document**:  
  The seller signs a deterministic JSON structure containing the ciphertext, IV, metadata, timestamp, and nonce. The `check()` operation validates this signature using the seller’s public key extracted from a CA‑signed certificate. Any modification to the ciphertext or signed metadata will cause signature verification to fail.

- **SR4 – Integrity and Auditability of the Sharing History**:  
  Each access list entry (sharing record) is individually signed and stored in the document, producing an auditable disclosure trail. In the current verification path, `check()` validates seller-signed entries and returns an `access_list_ok` flag plus detailed per-entry results. Extending verification to arbitrary sharers is a natural next step.

## 3. Conclusion

This project implements an end‑to‑end prototype for protecting, storing, and selectively disclosing transaction documents in a multi‑party ChainOfProduct scenario. At the cryptographic layer, we designed and implemented a custom secure document format and a Java library that combine AES‑256‑GCM, RSA‑OAEP, and digital signatures to provide confidentiality, integrity, controlled disclosure, and an auditable sharing history. At the infrastructure layer, we deployed a realistic multi‑VM environment with network isolation (front‑net/back‑net), a firewall configuration, TLS‑protected channels for both API↔client and app↔database traffic, and a dedicated Certificate Authority that issues X.509 certificates for all partners.

The current implementation covers the main mechanisms required to target SR1–SR4, with encrypted per-recipient access, seller-signed integrity checks, and signed sharing entries. At the same time, some aspects remain partial in the prototype, namely stronger replay protection and broader verification of non-seller sharing signatures.

The group‑based disclosure extension and Group Management Server show how the original design can be cleanly extended without changing the cryptographic core: groups are resolved dynamically, but access is still granted only by individual wrapped keys bound to validated certificates. In future work, we could further strengthen the system by adding mutual TLS between app and group‑server, implementing certificate revocation mechanisms, integrating hardware‑backed key storage for long‑term private keys, improving the user‑facing UI for key and certificate management, implementing a real registration and login functionality and further tightening the firewall rules for preventing DoS atacks. Overall, the project provided practical experience in designing secure data formats, implementing cryptographic protocols, and deploying a small but realistic PKI‑enabled infrastructures.
